# US-20.0 — Public Song Page & Anonymous Public Read

Demo of issue #250, re-run end-to-end against the final commit (`272378a`). Every
acceptance criterion below is backed by observed outcome — not "the page
rendered" and not "the tests are green".

## Setup

Throwaway stack: API on `:8077` (Mongo on `:27018`, local storage), web on
`:3077`. Three clips seeded for one owner (no clip-ingest API exists, so they go
in via Beanie + storage), plus a second account to act as a non-owner:

| Clip | Visibility | Note |
|---|---|---|
| Midnight Drive | **public** | the shared link; **derived from the private parent below** |
| Private Parent Sketch | private | ancestor of the public clip — exercises the ancestry redaction |
| Unreleased Sketch | private | the "private clips stay private" case |

Each carries a real 6-second WAV tone, title, style tags, lyrics, and a
generation recipe (`seed=4242`, `inference_steps=30`).

The public clip deliberately has a **private parent**, so the redaction is
exercised against real ancestry rather than an empty list.

## AC1 — signed-out visitor

Opened with **no session cookie** (`document.cookie` held only Next's HMR dev
hash). No redirect.

![Anonymous visitor viewing the public song page](us200-anon-public.png)

Playback is real, not inferred. The engine builds its element with `new Audio()`,
so it never enters the DOM and "an `<audio>` exists" would prove nothing. A
direct probe of the proxy URL in the signed-out browser:

```json
{ "duration": 6, "readyState": 4, "mediaError": null, "playError": null,
  "paused": false, "before": 0, "after": 1.46, "advanced": true }
```

The audio decodes to the correct 6-second duration and `currentTime` advances —
served anonymously. Seeking works too:

```
$ curl -sD- -H "Range: bytes=0-99" localhost:3077/api/clips/{id}/stream
HTTP/1.1 206 Partial Content
content-range: bytes 0-99/529244
```

## AC4 — ownership gating, and the redaction made visible

The owner's view of the **same URL**. Note the 4th (globe) icon and the
**Actions** dropdown that visitors don't get — and the **Generation history**
panel showing *"Derived from → Private Parent Sketch"*:

![Owner viewing their own public song page](us200-owner-public.png)

The signed-out visitor gets none of it. Same clip, same URL:

```json
{ "SIGNED_OUT": true, "title": "Midnight Drive",
  "hasGenerationHistory": false, "leaksParentTitle": false,
  "leaksParentId": false, "leaksOtherPrivateClip": false, "ownerControls": [] }
```

That's the ancestry redaction with a visible consequence: the owner sees the
lineage panel; a stranger gets no panel at all, and the private parent's ObjectId
appears **nowhere in the DOM**.

## AC5 — the two reads are distinct, and the redaction is asymmetric

Same endpoint, two callers:

```
anonymous → workspace_id: null,   seed: null, inference_steps: null, parent_clip_ids: [],          is_owner: false
owner     → workspace_id: "…8df", seed: 4242, inference_steps: 30,   parent_clip_ids: ["…8e0"],    is_owner: true
```

`user_id` is absent from both — ownership travels only as `is_owner`. The private
parent's id is confirmed absent from the anonymous payload.

Meanwhile the owner-scoped CRUD read is untouched: `GET /clips/{public}` still
401s anonymously and 404s for a non-owner (pinned by the unchanged CRUD test).

## Known limitations

- **Private-clip playback is still broken** (pre-existing, not a regression): an
  `<audio src>` cannot attach a Bearer token. Measured: the path `main` shipped
  (`/api/v1/clips/{id}/audio`) 404s, so playback was dead for *every* clip
  before this change. Public now 200s; private stays 404 → **#282**.
- **The public rate limiter collapses into one shared bucket** behind the BFF —
  all proxied traffic arrives as the Next server's IP → **#283**.
- **The access token is never refreshed** and `isAuthenticated` ignores `exp`, so
  owner-only pages break after expiry → **#285**.
- Authenticated non-owners lose Download on a public clip (the whole menu is
  hidden). Not a regression — the page didn't open for them at all before.
- Cover art still doesn't load (`clipArtworkUrl` has the same dead-path problem).
- `unlisted` deferred to US-20.7 (#219) — `Clip` has no such field.
- Related songs / lineage are owner-scoped authenticated reads, so they degrade
  to nothing for a visitor (by design — see AC4 above).
